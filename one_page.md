# 一、背景：关于 Alva

首先我粗浅地看了一下咱们的产品，在我理解中，Alva 的核心价值是：把机构级的投资能力个人化。

它体现在两层：

**第一层是：金融基础设施的平权。**

过去高质量数据源、量化分析工具、回测系统、自动化监控能力，主要属于机构投资者或有工程能力的专业交易者。Alva 把这些能力封装进自然语言工作流里，让普通/准专业投资者也能调用机构级工具，而不是只能看行情软件和碎片化资讯。

**第二层是：投研智能的平权。**

Alva agent 不只是帮用户查信息，而是让用户拥有一个可定制的投研团队：它可以持续搜集信息、验证逻辑、跟踪持仓、识别异常、生成界面，并在关键变化发生时主动提醒。用户不再只是被动接收信息，而是可以结合 Agent 的能力，不断把自己的投资判断系统化、自动化。

# 二、什么是 Playbook？

Playbook 像是一个 live investing app：前端是用户可消费的投资界面，后端由持续更新的 feed、计算逻辑、调度和 alert 支撑的“工作流”。它的价值不是展示一次性分析，而是把一个投资任务变成可持续运行、可检查的产品。

我注意到在 Alva 里，Playbook 是可以被发现、分享、订阅、remix 的。那么它定位，我理解是一个被用户在一定约束条件（Skill）下，充分结合了自己的理解、偏好 build 出来的，具备一定传播、复用、二次开发价值的产物。

# 三、什么是 Skill？

Skill 是一个可复用的能力单元，在这个作业的语境下，对我们这个 task 中的 portfolio-watch-skill 来说，它包含了用户在 build Playbook 时所需的“方法论 + 约束 + 可供使用的工具/能力集合 + 一些模板和开发范式”。这个 Skill 的存在，是为了让用户能在这一系列构建契约下，稳定地产出相对高质量的 Portfolio Watch Playbook。

# 四、portfolio-watch-skill 的边界应该在哪里？

如果需要服务的场景越宽广，我们越应该让 skill 更自由，施加更少的限制；如果需要服务的场景越具体，我们可以考虑在 skill 中注入更多的指引或约束。

我观察到在 Alva skill 里已经包含了大量的约束和可供使用的工具/能力集合。这些我打算照单全收，尽量保留。用得上的直接用，用不上的再说。

但是 Skill 中，方法论、模板、开发范式的部分，这里面的平衡怎么取，是我要想一想的：

- 哪些预设是必要的？
- 哪些自由度是应该留给用户的？
- 可复用是多大程度上的可复用呢？

虽然作业里面对复用性的要求，明确表示了说希望可以对没见过的持仓复用就行，但是我整体对skill的理解还是要尽量做到求同存异。在里面定义好什么是普适意义上的 Portfolio Watch，给一些有复用价值的示例，不再做过多的约束。（因为我觉得如果只想更换持仓的话，走 remix 也挺快，好像没必要用 skill 从头跑）

同时考虑到每个用户对什么算 big，该怎么 watch 的理解可能都不一样，所以我不希望把握个人的理解在 skill 层面强加给用户。涉及我个人理解的部分，会更多的通过我的 Playbook 来承载。（但是这一部分实际做出来还是限制的比较死，不过考虑到这个 skill 触发的场景也限制的比较死，我就没太继续纠结这一部分。）

# 五、这个 Playbook 里该有啥？

## 1. 用户要的是什么？

用户说的是：

> “keep an eye on my NVDA, TSLA, and AAPL, ping me when something big happens.”

用户并没有说“涨跌 5% 提醒我”，而是说 “something big”。这意味着用户期待系统替他判断：

- 什么值得看；
- 什么可以忽略；
- 什么时候需要打扰我；
- 为什么这件事重要。

所以这里的核心不是 ticker monitoring，而是 **attention delegation**。

## 2. 谁会说这样的话？

- “keep an eye on”，说明用户不是一个全天盯盘的人，不是那种 day trader
- “my NVDA, TSLA, and AAPL”，说明用户持有一定个股，而且个股占据了相当的仓位，不是一个纯指数/ETF投资者，而且是有持仓的，不是一个纯粹的 watch list
- “ping me”，说明用户不完全是一个被动投资者，TA需要结合情况做出判断，自己进行操作
- “big”，说明用户有一定的风险承受能力

因此我们这个 Playbook 的定位，应该是：

> **Low-noise review-trigger system for busy holders**

基于以上分析，我们这个 Playbook 的核心，不是做为一个大而全的、特别眼花缭乱的专业看板。能说出这种话，有这种诉求的用户，大概率不是一个有分析这种大量复杂信息意愿的人。所以我们这个看板的目标是解决这类用户的核心诉求，避免堆砌一堆他看不懂的指标和信息。

## 3. Playbook 里面的模块和逻辑

### 3.1 Portfolio Spec：把模糊输入变成可执行的监控范围

用户只需提供 ticker、watchlist 或持仓；Skill 会补齐必要的默认值，包括权重、基准、刷新频率和提醒敏感度。

如果用户没有提供权重，系统使用等权重作为监控假设，并在界面和 README 中明确说明；没有成本价或账户数据时，不推断盈亏或已实现收益。默认使用 SPY 作为相对表现和异常判断的主基准，并将 QQQ 用作图表比较基准。

这样做的目标是：不因用户输入不完整而卡住，但也不把假设伪装成事实。

### 3.2 Data Layer：从价格到事件上下文建立证据

每次刷新会为每个持仓拉取并计算以下信息：

- 价格表现：1D、1W、1M 收益和归一化价格路径；
- 相对表现：相对 SPY 的当日超额涨跌；
- 成交量：当日成交量相对过去 20 日中位数；
- 趋势与波动：MA20、MA60、RSI、当日波动相对历史正常水平；
- 组合影响：个股涨跌乘以其监控权重；
- 事件上下文：近期新闻、财报日期、分析师目标价或预期变化。

价格和技术数据负责发现异常；事件数据负责帮助解释异常。页面只会展示当前实际接入并成功获取的数据，也会明确说明未覆盖的范围。

### 3.3 Attention Engine：判断“值得看”与“值得打扰”

核心判断不是“涨跌超过 X% 就提醒”，而是综合判断：

> **Attention = 异常程度 + 组合相关性 + 可解释性 + 事件上下文 − 重复噪声**

异常程度主要看价格相对自身历史波动是否异常、相对 SPY 是否显著偏离、成交量是否放大、趋势是否拉伸，以及日内波动是否扩大。组合相关性强调权重：大仓位的中等波动，可能比小仓位的大幅波动更值得优先关注。

系统将结果翻译成三种用户可理解的状态：

- 绿色「无需关注」：没有足够显著的变化；
- 黄色「留意一下」：有值得阅读的变化，但不值得打断用户；
- 红色「请立即关注」：异常程度高、组合影响达到中断门槛，并且有足够证据或异常影响特别大。

这里最重要的设计是把“页面里值得看”和“手机上值得打扰”分成两道门槛。黄色信号是有价值的信息，但不应成为通知噪声。

### 3.4 Status-first Interface：先回答，再给证据

界面不是从指标表开始，而是先回答用户最关心的问题：Anything big?

第一屏包含：

- 一句基于当前数据生成的组合总结；
- 每只持仓的红黄绿状态；
- 1D 涨跌、相对 SPY、成交量比例和组合影响；
- 已匹配的新闻或事件链接；
- 最近一次数据更新时间。

用户先得到结论，再决定是否继续下钻。趋势图放在第一屏之后，作为组合、个股与 SPY/QQQ 的证据层；用户可切换 Portfolio、Tickers 和 Compare，以及 1M、3M、6M、All 等观察区间。

### 3.5 Evidence & Transparency：让每个判断可解释

在证据明细中，系统将持仓按优先级排序，并展示：

- 发生了什么；
- 为什么这次变化异常；
- 它对组合的影响；
- 价格、成交量、趋势和波动的证据；
- 已匹配的事件证据；
- 数据盲区和未接入能力。

因此用户不需要相信一个黑盒分数，而能看到系统为什么把某只股票判为黄色或红色。

### 3.6 Alert Loop：默认安静，只在必要时触达

Playbook 在美股收盘后按工作日运行。绿色和黄色信号只留在页面中；只有红色、非重复、达到组合影响门槛的变化才有资格发出 Telegram 通知。

通知中会包含受影响的持仓、简短原因、组合影响，以及跳回 Playbook 对应证据位置的链接。系统同时记录本次为什么发出、为什么不发出，避免用户把“没收到通知”误解为“系统没有运行”。
此外，系统支持一次性设置确认和手动测试通知：它们只用于验证通知链路，不会被伪装成市场信号。
